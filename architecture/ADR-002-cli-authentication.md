# ADR-002: CLI Authentication and Signed Approvals

## Status
Proposed / Accepted (Sprint 5)

## Context
The previous implementation of the `agt escalate approve` command relied on a user-supplied privilege ring flag, which posed a significant escalation vulnerability. Furthermore, there was no mechanism to verify the identity of the approver in a distributed environment.

## Decisions

### 1. Agent Registry
- **Mechanism:** Introduced the `AgentRegistry` trait to store and retrieve agent public keys and their assigned `PrivilegeRing`.
- **Source of Truth:** The ring of an approver is now looked up from the registry via their DID, rather than being accepted as a CLI flag.

### 2. Signed Escalation Approvals
- **Mechanism:** `EscalationManager::approve` now requires a cryptographic signature generated by the approver's private key.
- **Verification:** The signature is verified against the approver's public key retrieved from the `AgentRegistry`.
- **Payload Canonicalization:** A deterministic `approval_payload` function is defined in `escalation.rs` to ensure consistent signature generation and verification across different nodes and CLI versions.

### 3. Payload Format
- **Format:** `format!("{}:{}:{}", event_id, agent_did, requested_ring)`
- **Components:**
  - `event_id`: The unique Uuid of the escalation request.
  - `agent_did`: The DID of the agent requesting escalation.
  - `requested_ring`: The privilege level being requested.

### 4. CLI Authentication
- **Mechanism:** The `agt escalate approve` command now requires a `--key-dir` flag and uses the `AGT_KEY_PASSPHRASE` to recover the approver's private key for signing.

## Rationale
- **Registry Lookup** ensures that privilege levels are managed centrally and cannot be spoofed by CLI callers.
- **Digital Signatures** provide non-repudiation and ensure that only the authorized holder of a System-ring private key can approve sensitive privilege transitions.
- **Canonical Payload** prevents "signature mismatch" bugs that often arise from non-deterministic serialization formats.

## Deferred / Out of Scope
- **Registry Hardening:** The current `MemoryAgentRegistry` is a local mock. Distributed registry implementations (e.g., via a consensus layer or signed attestations) are planned for the 90-day horizon.
- **Consensus-based Approval:** Requiring multiple signatures for the highest-privilege transitions (e.g., to `System` ring).

## Security Considerations
- The security of this scheme depends on the integrity of the `AgentRegistry`. If an attacker compromises the registry, they can swap a valid public key for their own and sign malicious approvals.
- For Sprint 5, the registry is considered within the trust boundary (e.g., running as a protected service). Hardening the registry-to-manager communication is a 90-day target.
